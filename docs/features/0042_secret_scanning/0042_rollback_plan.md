# 0042 — Rollback Plan

This feature ships only an additive Python skill module
(`agents/cwe/cwe_agent/skills/secret_scan/`). It does not modify any
existing skill, schema, env-var contract, or runtime model. Rollback
is correspondingly trivial.

## Pre-flight

Note which phases are live in the deployment you're rolling back. The
six phases are:

1. PEM block detector
2. Cloud / SaaS provider patterns
3. Crypto wallet detection
4. Substrate detection
5. Config-file scanning + entropy fallback
6. SKILLS.md + boundary docs

## Rollback by phase (newest first)

### Phase 6 — Documentation

- Revert `SKILLS.md` changes.
- Revert any operator-doc updates referencing the new skill.

### Phase 5 — Config-file scanning + entropy

- Delete `secret_scan/config_files.py` and `secret_scan/entropy.py`.
- Remove `--exclude-rules entropy_generic` plumbing if any was added
  to the CLI.
- The remaining sub-modules (PEM, cloud, crypto, substrate) keep
  working unchanged.

### Phase 4 — Substrate detection

- Delete `secret_scan/substrate.py`.
- Drop the corresponding test file.
- Other phases unchanged.

### Phase 3 — Crypto wallet detection

- Delete `secret_scan/crypto_wallets.py` and `data/bip39_english.txt`.
- Drop test files.
- Phases 1+2 (PEM + cloud) keep working.

### Phase 2 — Cloud / SaaS providers

- Delete `secret_scan/cloud_providers.py`.
- Phase 1 (PEM) keeps working.

### Phase 1 — PEM blocks

- Delete the entire `agents/cwe/cwe_agent/skills/secret_scan/` package.
- Remove the `check_secrets` tool registration in the CWE agent.
- Restore the previous extension list in `file_scanner.py` (no change
  was made — the extension override is per-skill, internal to
  secret_scan).

## Full rollback

```bash
git revert <0042 commits in reverse order>
```

No DB schema changes. No application binaries changed (skill is added
as a Python module loaded at agent start). No env-var contracts
introduced or removed beyond the optional
`VULTURE_SECRET_SCAN_ENTROPY` (default off; ignored if not set).

## What this rollback does NOT do

- Doesn't touch `auth_check.py` (which still owns the
  `name = "value"`-shape detection for CWE-798).
- Doesn't touch `info_exposure_check.py` (cleartext storage).
- Doesn't touch the catalog detector or any other CWE skill.
- Doesn't change the file-scanner's default `CODE_EXTENSIONS` for
  other skills — the extension override was per-skill.
- Doesn't change findings recorded in `audit_memories` from prior
  runs. The DB rows persist; only newly-generated findings stop being
  emitted.

## Post-rollback gap

After full rollback, the original audit gap returns:

- No PEM-block detection.
- No cloud / SaaS provider content-pattern detection.
- No cryptocurrency wallet secret detection.
- No Substrate-specific detection.
- No `.pem`/`.key`/`.crt`/`.env` file scanning.
- CWE-798 coverage falls back to the 5-pattern name-based detector
  in `auth_check.py:24-31` (which was the pre-0042 baseline).

If you rolled back due to false-positive noise from the entropy
detector, prefer rolling back **only Phase 5** (entropy + config-files)
rather than the whole feature — the cloud + PEM + crypto detectors
have very low FP rates and shouldn't need rolling back together.

## Risk-assessment matrix

| What rolls back | Affects | Operator-visible change |
|---|---|---|
| PEM detector | New PEM-block findings stop appearing | Existing PEM findings in DB remain; auditor stops detecting new ones |
| Cloud patterns | ~50 provider patterns gone | Findings count drops; named-LHS detection still works |
| Crypto / Substrate | BIP-39 / BIP-32 / WIF / ETH / Polkadot detection gone | Crypto-asset secret leaks become invisible again |
| Entropy + config-files | `low`-severity entropy noise gone (welcome); JSON/YAML config-secret detection gone (not welcome) | Operator should rebenefit from JSON config detection by re-shipping Phase 5 with entropy disabled by default |

## Schema-level rollback

Feature 0042 does **not** change any database schema. There is no
migration to roll back. The only persistent footprint is the
`findings` rows generated by the new skill, which carry the same
shape as findings from any other skill.

If you want to also remove the historical findings from prior runs
(e.g., to suppress old false positives now that the skill is gone):

```sql
DELETE FROM findings
WHERE check_id LIKE 'cwe.secret_scan.%';
```

This is optional — leaving them is harmless.
